Resource_system

Ресурсы и данные

Данные

Основные положения

  • Первичный ключ является int identity (autocounter)
  • Каждый объект имеет дополнительно user-defined name (UserName) и ссылку на логический folder (FolderID). Отдельно присутствует табличка с folder’ами.

 

1) Используемые структуры данных описываются в .XSD файлах, которые в последующем считываются при помощи MS XML 4.0. Эти файлы сложены в отдельной директории. Один из них описывает общие для всех структур "простые типы" (вектора, цвет, ссылка на другую описанную структуру, ссылки на все бинарные ресурсы), а каждый из остальных описывает свои структуры данных, пользуясь этими "простыми типами" или другими описанными сложными структурами. Значения полей по умолчанию проставляются как атрибут "default". При работе с конкретным типом структуры в кэш MS XML считываются все .XSD файлы.

 

2) Существует утилита, которая считывает информацию из указанного ей .XSD файла, и преобразует его информацию в таблицы MS SQL и С++ заголовки с описанием структур и прототипами их функций. Также она создает сами функции (в .cpp или в .h файле) (в частности, функции бинарной сериализации).

 

3) В .XSD файлах при описании структур используется наследование, которое впоследствии отобрадается на физическую структуру таблиц в БД и на сгенерённые структуры данных.

 

Преобразование в таблицы происходит следующим образом (все ссылки в таблицах имеют тип int):

 

  • для всех узлов структуры с числом вхождений <= 1 (либо, если оно известно заранее) просто создаются колонки с именами, соответствующими именам XSD элементов
  • если узел может появляться более одного раза, то для структур, в нем содержащихся, создается отдельная таблица, помимо прочих полей содержащая поле с ссылкой на запись в основной таблице (это правило действует рекурсивно)
  • поля, имеющие ссылочный тип, преобразуются в ссылки на соответствующие таблицы.
  • если некоторые поля структуры определены в базовом классе, то вместо этих полей в таблице появляется поле – ссылка на таблицу с объектами базового класса (правило действует рекурсивно)

 

Преобразование в .CPP/.H файлы:

 

  • каждый уровень дерева преобразуется в соответствующую структуру
  • для элементов, которые могут входить несколько раз, создаются массивы (vector)
  • если элемент является ссылкой на другой ресурс (табличный), то для него заводится поле типа CDBPtr<Type>, куда пишется идентификатор ресурса
  • если элемент является ссылкой на бинарный ресурс, то для него заводится строка с именем ресурса
  • описываются функции бинарной сериализации, chunk_id для элементов берется из аттрибутов элементов в .XSD
  • описываются функции получения контрольной суммы; в расчет берутся все помеченные соответствующим образом поля. Конторольная сумма нужна для проверки синхронности данных для игры по сети peer-to-peer

 

После первичной обработки .XSD файла он записывается в БД. В дальнейшем при повторной обработке этого файла поступивший на вход файл сравнивается с предыдущей версией в БД. В этом случае утилита изменяет структуру БД в соответствии с произведенными изменениями (удаляются колонки и таблицы, соответствующие удаленным элементам, добавляются для вновь появившихся; старые данные при этом удаляются из БД только если удалено соответствующее поле в .XSD файле). Для каждого объекта помимо его основной информации в БД хранится также строка, идентифицирующая объект в программе (то, что показывается в редакторе) и дата его последнего изменения.

 

4) Редактор анализирует .XSD файлы из БД, и на основе полученной информации предоставляет интерфейс редактирования структур данных в БД в виде property control.

 

Все внесенные изменения отображаются в property control и записываются в базу (есть идея не записывать изменения сразу в базу, а складывать их в очередь SQL-запросов пока юзер на нажмет на Save, что ведет к последовательному исполнению запросов и очистке очереди запросов). Редактор умеет записывать объекты из БД в бинарном формате, имитируя функцию бинарной сериализации для соответствующей структуры (на основе .XSD файла).

 

Экспорт производится путём выбора объектов и/или веток деревьев в соответствующих закладках. Под объектом подразумевается текстуры, модели, звуки, танчики, пехотинцы, карты, кампании, и т.д. При экспорте проверяется дата последнего экспорта (для избежания экспорта up-to-date объектов). Проверку можно принудительно подавить (forced export). Планируется возможность рекурсивного экспорта (заэкспортить Т-34 с моделькой, текстурой, звуками, эффектами, статсами и т.д.). Можно указать дату последнего изменения, которая будет являться нижней границей для экспортируемых структур (при этом все ссылки структуры по прежнему обрабатываются). Есть возможность зажать все экспортируемые файлы в PAK.

 

5) Игра. Считывает данные из запакованных бинарников. Бинарники есть трех типов:

  • основные, загружаются первыми в произвольном порядке; не должны содержать ресурсов с одинаковыми ключами
  • патч; помимо прочей информации содержит номер версии, после основных бинарников загружаются все патчи в порядке возрастания версии, заменяя ресурсы
  • мод; может быть подключен после загрузки основного бинарника и патчей, для всех замещенных объектов записывается оригинальное состояние, впоследствии может быть отключен
  • сейчас поддерживается только один мод в один момент времени, но систему можно легко расширить до множества модов (как показал опыт а7 при множестве модов становится неочевидным пользовательский интерфейс менеджмента модов и встают в полный рост сопутствующие проблемы (хранение save’ов, сделанных под этими модами; сгенерённые карты и т.д.))

 

Бинарник представляет из себя записанный массив с ключами ресурсов, их типами и самими данными ресурсов. Тип бинарника определяется по его расширению, у пользователей в редакторе должна быть отключена возможность делать что-то, кроме модов.

 

При загрузке система ресурсов считывает ключи и типы ресурсов, сами объекты подгружаются при непосредственном запросе к их данным. Все структуры игры (за исключением самой системы ресурсов) должны работать с ресурсами данных исключительно через CDBPtr. Система ресурсов предостваляет интерфейс для получения const указателя на ресурс по ключу-строке, который в дальнейшем должен хранится как CDBPtr. При его сериализации в файл записывается только ключ ресурса.

Первичный ключ в базе – int identity.

Бинарные ресурсы

Бинарные данные есть двух типов – те, что интересны пользователям для редактирования внешними тулзами (текстуры, звуки – «открытые» ресурсы) и те, что нельзя редактировать внешними тулзами (модели, анимации, и т.д. – «закрытые» ресурсы). «Открытые» ресурсы при экспорте раскладываются по каталогам с красивыми именами (так же, как и их исходники), а «закрытые» сваливаются в одну директорию (например, “binary”) и именуются DB’шными ID’шниками.

Пакуется всё zip’ом (методы STORE или DEFLATE). Файловая система поддерживает как файлы в открытом виде, так и запакованные в zip прозрачно для пользователя. Если существует несколько одинаковых файлов в разных zip’ах и один в открытом виде, то среди них выбирается файл с самой последней датой. При этом, если подключен мод, то сначала файл ищется в моде (по тем же правилам), а только потом в основном ресурсе.

Тексты живут в базе и при экспорте либо записываются в один файл (бынарный массив wstring’ов) либо считаются «закрытыми» данными и записываются в одну директорию с именами по DB’шным ID’шникам.

 

Не до конца определено с модами.
