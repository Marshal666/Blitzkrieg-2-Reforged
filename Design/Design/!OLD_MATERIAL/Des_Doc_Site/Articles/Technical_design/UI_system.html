UI_system

User Interface System 

Система интерфейсов.

 

Что хотим:

  1. сделать так, чтобы не заниматься вбиванием текстур и расставлением координат ручками.
  2. чтобы дизайнеру можно было создать библиотеку контроллов – для конкретной игры и применять ее.
  3. чтобы создание контролла с новым поведением было простым делом и не требовало присутствия программиста.
  4. нужно визуальное редактирование.
  5. часть работы дизайнера стыковалась с частью работы программера максимально просто.

 

Что предлагается.

 

Нужен редактор интерфейсов, в котором дизайнер видит картинку, такую же как в игре.

 

Расположение элементов.

У дизайнера есть базовый набор элементов (окно с текстом, кнопка, список и т.д.), который он может расширять:

Создать кнопку «OK», специфичную для игры и т.п.

Например, что когда дизайнер хочет, чтобы стиль сохранялся, он использует 1 кнопку на все экраны. Если нужно изменить какой-то часто встречающийся элемент – то это делается в 1 месте.

 

Логика взаимодействия элементов.

В большой части случаев действия, которые нужно произвести при получении какого-то сообщения однообразны и хорошо формализуются. Например, пришло сообщение о нажатии кнопки, нам нужно показать окно, выставить флажок, послать другое сообщение. Какую логику тоже можно вынести в редактор.

Нечасто встречающиеся случаи сложных действий, напр. сгенерить список – нужно договариваться с программером.

 

Внутреннее строение элемента интерфейса.

 

Shared – Personal data.

Элемент интерфейса состоит из общих данных этого контролла (например текстура) и частных данных instance этого контролла (тултип).

 

Window = Effect + Background + Foreground.

 

У элемента интерфейса есть background, который может быть простым (текстура), сложным (tiled-текстура) и т.п.

Foreground – то, что рисуется поверх, например текст.

Effect – то, что влияет на вывод элемента на экран, например мигание, выезд за пределы экрана.

Преимущество разделения в том, что контролл не должен знать об эффектах, легко добавлять эффекты, не нужно при этом менять контролл.

По умолчанию Segment у контролла не вызывается, контролл или его эффект подписывается на Segment если им это нужно.

 

Обработка сообщений.

 

Сообщения, на мой взгляд делятся на 2 типа

- нужные только 1 окну

- нужные всем окнам без исключения

Предлагаю их так и разделить, на сообщения 1 типа каждый контролл подписывается, сообщения 2 типа рассылаются всем.

ID у контроллов – текстовые, ID сообщений – тоже текстовые.

 

Прочие фичи.

 

  1. Copy /Paste из/в clipboard.
  2. Drag & Droр (windows-like?, diablo-like?)
  3. 3D model view control
  4. mini-html text formatting from a5.

 

Система UI должна быть максимально независимой от особенностей реализации игры и графической сцены и при этом должна быть максимально шустрой.
